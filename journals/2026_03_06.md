	- ## **Discussed**
		- ### **1. Germany E‑Invoicing Requirements**
			- Germany requires invoices to be issued in:
				- **PDF/A‑3** format with strict structural and font requirements. [[260306_001-normal | Word]](https://ingrammicro-my.sharepoint.com/personal/tim_huygen_ingrammicro_com/_layouts/15/Doc.aspx?sourcedoc=%7B135E067A-6383-4245-AC4C-A6D9547E6412%7D&file=260306_001-normal.docx&action=default&mobileredirect=true)

[[260306_001-normal | Word]](https://ingrammicro-my.sharepoint.com/personal/tim_huygen_ingrammicro_com/_layouts/15/Doc.aspx?sourcedoc=%7B135E067A-6383-4245-AC4C-A6D9547E6412%7D&file=260306_001-normal.docx&action=default&mobileredirect=true)
			- The current tool:
				- Is outdated and no longer fully supports PDF/A‑3.
				- Can embed XML, but cannot meet all strict PDF/A‑3 validation rules.
		- ### **2. Current Validation Findings**
			- Multiple external validators exist; Germany did **not** mandate a single official validator.
			- The validator provided by the **German team (Christian)** successfully validates all our invoices.
			- Some other validators fail due to strict PDF/A‑3 rules (font dictionaries, rendering metadata, etc.).
			- German authorities have not yet decided which validator they will use.
			- Conclusion:
				- If the invoice passes the validator provided by Germany’s team, it is considered acceptable for release at this stage.
		- ### **3. XML Portion Fully Valid**
			- The XML side of the invoice passes all required validation rules.
			- All remaining issues are tied to **PDF formatting restrictions**, not invoice data.
		- ### **4. Release Strategy Prior to SAP PS1 Migration**
			- Germany will eventually migrate to **SAP PS1 (XPS 3.0)**, which includes SAP’s e‑invoicing/Digital Reporting Cockpit (DRC).
			- Until then, the current approach is a temporary compliance layer.
			- Therefore:
				- The current PDF/XML solution is sufficient for a **controlled pilot**, provided the German team validates it.
		- ### **5. Need for Replacing the Legacy PDF Tool**
			- Independent of Germany’s requirements, the current tool:
				- Struggles with large invoices.
				- Slows down invoice generation.
				- Cannot achieve full PDF/A‑3 compliance.
			- A new invoice generation tool will be required (separate epic).
		- ### **6. Communication Approach**
			- Rina asked whether resellers should be told that strict validators may fail.
			- Tim advises **not to introduce unnecessary doubt**, and instead:
				- Communicate that validation was performed using the validator recommended by the German team.
				- This validator confirms the invoices are compliant with current expectations.
		- ### **7. Rollout Flow**
			- Engineering resolves remaining issues.
			- Generate final sample invoices.
			- Validate with Christian (German team).
			- Perform pilot with a selected reseller.
			- If accepted, roll out generally.
