	- ## **Discussed**
		- ### **1. Background: Why the Test Repository Is Being Reintroduced**
			- Previously, Jay required teams to maintain an **Excel-based UAT repository** for:
				- Test cases discovered outside QA
				- Additional business scenarios identified during UAT
			- With Navigator & JIRA becoming central systems, Excel tracking is obsolete.
			- Jay now requires a **centralized repository in JIRA**, enabling:
				- Reuse of prior test cases across features
				- Better governance of QA/UAT coverage
				- A source of truth for test scenarios across domains
	- ---
		- ### **2. Purpose of the JIRA Test Repository**
		- The repository should allow:
			- #### **Capabilities Team**
				- Retrieve **business test scenarios** already known for a domain before asking business again
				- Combine existing repository test cases with new ones from business
				- Avoid missing critical scenarios in IST/UAT
				- Increase consistency across releases
			- #### **Deployment Team**
				- Ensure test cases created by deployment include:
					- Required metadata (labels/tags)
					- Clean structure for future reuse
				- Support QA when they review missing or regression-relevant scenarios
			- #### **QA / IST Team**
				- Pull reusable test keys into their IST dashboards
				- Identify regression candidates for repeated execution
				- Avoid test case drift across releases
	- ---
		- ### **3. Structure of the Repository (Confluence + JIRA)**
			- Meet and Annabelle developed a **Confluence page** documenting structure and usage rules.
			- Each **Test Key** (test case) must contain mandatory metadata for filtering:
				- #### **A. Mandatory Filters**
					- **Country**
						- Allows capabilities/deployment to filter per region (US, EMEA, LATAM)
					- **Priority**
						- Only Critical / Medium cases should be reused
						- Low/cosmetic cases excluded
					- **Component (Domain)**
						- Pricing, Subscriptions, X4C, TSO, etc.
						- Essential for repository filtering
						  
						  All agreed these are necessary.
				- #### **B. Vendor-Specific Tag**
					- Many test cases are **vendor-specific** (MSFT, Adobe, Proofpoint, IBM).
					- Must be labeled **Vendor Specific = Yes/No**.
					- If *Yes*, must add vendor label (e.g., `Proofpoint`, `Adobe`, `Microsoft`).
					- Deployment team must maintain this moving forward.
					  
					  Tim confirmed vendor-specific relevance is **high for subscriptions domain**.
				- #### **C. Regression Candidate Tag**
					- QA requested ability to mark test cases that **must always run** for a vendor or domain.
					- If `Regression Candidate = Yes`, that test case must be reused in every related feature’s IST/UAT.
					- Helps avoid missed coverage.
	- ---
		- ### **4. Naming Convention for Test Keys**
			- Required pattern:
			  
			  ```
			  <Component> – <Country> – <Feature Name> – <Description>
			  ```
			  
			  Example:
			  
			  ```
			  X4C – Brazil – Unite – Discount Calculation for Renewals
			  ```
			  
			  Reason:
			- Avoids hundreds of generic names like “X4C Test Case 1”
			- Enables quick filtering and identification
		- ---
		- ### **5. IST / UAT Workflow Integration**
		- How repository fits into existing process:
			- #### **Capabilities**
				- Retrieve vendor/domain test cases from repository
				- Share with Product/Business during requirements validation
				- Add new business scenarios if missing
				- Hand off to QA / IST
			- #### **Deployment**
				- Create test keys for UAT using repository rules
				- Ensure all metadata and labels are added
				- QA imports needed cases into IST dashboards
				- Deployment adjusts or supplements as needed for UAT
	- ---
		- ### **6. Key Clarifications Raised**
			- #### **Tim’s Points**
				- Subscription domain already has many reusable test cases
				- Questions whether test cases need retroactive tagging
				  → **Decision:** No retroactive updates required
				  → Only **future** test cases must follow new format
			- #### **Capabilities**
				- Need clarity on how existing test cases are discovered & used
				- Confirmed repository is **search-and-select**, not backward migration work
			- #### **IST Ownership**
				- IST team produces detailed Excel test cases today
				- These remain as inputs but repository will provide the cross-feature reusable layer
			- #### **Business Test Case Ownership**
				- Capabilities may no longer need to collect Excel lists from business
				- Navigator use cases + IST team’s test cases should become primary sources

			- If BIT starts using *global end user ID*, then:

					- Misaligned invoice references
					- Downstream billing disputes

					- Customers rely on the **CMP End Customer ID (XVS end user ID)**.
					- A sudden change is **not acceptable**.
					- Global end user ID is **not meaningful** for customers.
			- **Decision:**
				- **XVS will continue using CMP/XVS End Customer ID**, not the global one.
				- Subscription feed will contain global ID (used only for tax logic).
				- BIT will *not* replace XVS ID for now.

		- ### **3. Usage File Currency Handling**
			- Current behavior:
				- Unprocessed usage records **do NOT contain currency**.
				- Currency is specified in the **usage report object** created in Connect, not in the file.
				- Connect augments processed files, adding currency in processed outputs.
			- Team alignment:
				- To handle AWS needs, we must **split usage files by marketplace & currency**:
					- One file for currency different from local currency
					- One file for local currency
					- Each with separate Connect usage report objects
				- Additional field suggested:
					- vdoc field → identifies AWS entity (US, CA, etc.)
					- This requires Connector changes but **not** Connect Platform core changes.
			- ---
		- ### **4. XVS Impact – New Scenario**
			- Today, XVS supports **one vendor / one currency** per marketplace.
			- AWS marketplace introduces a **new scenario**:
				- Standard IaaS consumption → billed in customer account currency
				- Marketplace third‑party consumption → billed in  **3rd party vendor currency only**
			- This requires:
				- XVS rating engine updates
				- XVS BIT generation handling dual currencies
				- Ensuring correct mapping between USD BITs and CAD BITs
			- ---
		- ### **5. Marketplace Billing Rules (Key AWS Requirement)**
		- AWS rules:
		  | Billing Type | Currency Rule |
		  | **Standard AWS services** | Billed in **payer account currency** (USD or CAD) |
		  | **Marketplace services** | Always billed in the **3rd party vendor’s currency** |
		- Implications:
			- For marketplace charges, reseller must always be billed in **AWS currency**.
			- Even if reseller chooses to receive normal AWS consumption in CAD, AWS marketplace charges will **always** be 3rd party vendor currency.
			- If reseller does not have a 3rd party vendor currency account:
				- A 3rd party vendor currency reseller account must be created automatically for marketplace BITs.
			- Tim noted:
				- In EMEA this becomes more complex: marketplace sellers may operate in GBP/EUR, etc., so AWS may invoice us in more 3rd party vendor currencies.

- # Meeting [[Project/XVS rounding logic Optimization]]
	- ## **Agenda**
		- Review current decimal‑precision behavior in XVS/BSS and how it leads to invoice discrepancies.
		- Explore differences between vendor inputs (2–13 decimals) and platform calculations (8 decimals).
		- Discuss options for how to display duration, prorations, discounts, and unit prices in UI and invoices.
		- Define requirements for meeting **e‑invoicing validation standards** (e.g., Germany).
		- Align on next steps to determine a unified rounding and display strategy.
		  
		  ---
	- ## **Attendees**
		- Tim
		- Chris
		- Iaroslav
		- Irena
		- ---
	- ## **Discussed**
		- ### **1. Current Calculation Logic & Root Cause of Discrepancies**
			- XVS/BSS performs calculations using **8 decimal places** for key elements:
				- Duration
				- Discount amounts
				- Unit price multipliers
				- Microsoft Azure usage (up to **13 decimals**)
			- XVS
				- UI and invoices **round values to 2 decimals**, which creates:
					- Visible differences between displayed values (UI/invoice) and actual computed totals.
					- Cases where **qty × unit price ≠ total**, causing customer confusion.
				- Example:
					- Duration stored as **0.45193678** but shown as **0.45** → mismatched totals.
					  
					  ---
		- ### **2. Vendor Behavior (Microsoft / Adobe)**
			- Microsoft usage-based services (Azure) provide prices with up to **13 decimal places**.
			- Some Microsoft recurring services give only **2 decimals**.
			- Adobe also sends certain values at 2 decimals.
			- Because XVS calculates at 8 decimals, but UI displays 2 decimals, discrepancies are unavoidable.
			  
			  ---
		- ### **3. Double Rounding Problem**
			- Today the system rounds in both:
				- **BSS** (internal)
				- **PDF invoice generation**
			- This leads to “double rounding,” amplifying the discrepancy.
			  
			  ---
		- ### **4. Core Problem for Customers**
			- Customers expect **unit price × quantity = total**.
			- But because prorations and discounts use more precision than what is shown:
				- Totals don’t match customer calculations.
				- Large user counts (e.g., >20,000 seats) cause variances up to **£25 or more**.
				  
				  ---
		- ### **5. Tim’s Proposal: Show Discounted Reseller Unit Price**
			- Tim suggests a structural change:
				- Compute everything with full precision (6–8 decimals).
				- Then **round only the final discounted reseller unit price to 2 decimals**.
				- Invoice displays:
					- Standard unit price
					- **Discounted reseller unit price** (2 decimals)
					- Quantity
				- Total = discounted reseller unit price × quantity → **guaranteed match**.
				- This also solves:
					- Customer confusion
					- Support tickets
					- Inconsistent invoice reporting
					  
					  ---
		- ### **6. Alternative Option: Show Full Precision (6–8 decimals)**
		- Iaroslav & team suggest:
			- Display full‑precision values (e.g., 8 decimals) in UI and invoices.
			- This ensures all calculations pass strict e‑invoice validators (Germany, France).
			- But this approach:
				- May confuse customers (too many decimals)
				- Requires UI and invoice redesign
				- Deviates from current CMP behavior (2 decimals)
				  
				  ---
		- ### **7. Upcoming E‑Invoicing Requirements (Germany, France)**
			- E‑invoice validators (ZUGFeRD, Factur-X, etc.) follow strict rules:
				- **Unit price may have many decimals (up to 8)**
				- **Totals MUST be exactly 2 decimals**
			- If displayed prorations or unit prices don’t align mathematically, validators will **reject invoices automatically**.
			- Because automated validation replaces human interpretation:
				- Prior “manual explanations to customers” are no longer possible.
				- We must standardize rounding and display logic.
				  
				  ---
		- ### **8. Two Acceptable Technical Solutions**
			- #### **Option A — Store & Display Full Precision**
				- **User based pricing :**
					- Use 8–13 decimals for duration in:
						- UI (duration)
					- Standard reseller Unit price * duration (8 digits) rounded at 2 digits = Prorated Std Reseller Unit price (2 digits)
					- Prorated Std Reseller Unit price (2 digits) - discount = Disc Prorated Reseller Unit Price (2 digits)
					- **Total Disc Prorated Reseller price** (2 digits) = Disc Prorated Reseller Unit Price (2 digits)* Qty
					- Disc (Prorated Reseller) Unit Price has to be a column on the invoice.
				- **Consumption based pricing :**
					- Use 8-13 decimals
					- Std Reseller Unit price (8 digits) * duration (8 digits) = (Prorated) Std Reseller Unit price (8 digits)
					- Prorated Std Reseller Unit Price (8 digits) - discount = Disc (Prorated) Reseller Unit Price (8 digits)
					- **Total Disc (Prorated) Reseller Price** (2 digits) = Disc (Prorated) Reseller Unit Price (8 digits) * Qty
					-
			- #### **Option B — Round Inputs to 2 Decimals and Recalculate**
				- Round unit price and pro‑ration before calculations.
				- Produce exact invoice totals every time.
				- Cons:
					- Could diverge from vendor billing (especially Azure)
					- Risk of overcharging customers
					- Higher financial risk for Ingram at scale.
			- Tim recommends **Option A with a clear discounted unit price**.
		- ---
		- ### **9. Critical Requirement**
			- If prorations are shown as **0.45** but calculated as **0.451976**, then **customers cannot reconcile invoices**, making large enterprises escalate regularly.
			- To fix this, we must either:
				- Display the real precision for duration, **or**
				- Round all calculations to what is visually shown.
		- ---
	- ## **Next Steps**
		- Engineering to prepare **examples using both approaches**:
			- Full precision displayed (8 decimals)
			- Discounted reseller unit price approach
		- Compare invoice results, validator outcomes, and alignment with vendor pricing.
