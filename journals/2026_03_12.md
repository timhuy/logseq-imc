		- Maxim

		  ---
	- ## **Discussed**
		- ### **1. Harmonization Model – Confirmation**
			- Shariq confirmed the following critical points:
				- Marketplace can have **multiple CMP End‑Customer IDs** for a single real end customer (different physical locations).
				- In the **End‑User Engine**, these are harmonized into:
					- **One high‑level harmonized End‑Customer ID**, with
					- **Multiple Location IDs**, each mapped *1:1* to a CMP End‑Customer ID.
				- This means:
					- If an end customer has 5 locations → XVS will still display **5 tiles** during end‑customer selection (one per location).
					- Each tile represents a **unique CMP End‑Customer ID** and a **unique Location ID** in EUE.
					  
					  ---
		- ### **2. Impact for XVS and SAP**
			- Two architectural approaches were discussed:
				- #### **Option A — XVS fetches Location ID from End‑User Engine**
					- XVS calls End User Engine :
						- Input = CMP End‑Customer ID
						- Output = harmonized Location ID
					- XVS passes that Location ID to SAP.
				- #### **Option B — SAP calls End‑User Engine directly**
					- XVS only passes CMP End‑Customer ID.
					- SAP is responsible for calling EUE to translate CMP ID → Location ID → state → tax.
			- Shariq confirms **both options are possible**, but SAP currently is **not yet integrated** with EUE.
			- For now:
				- **XVS will handle calling EUE and providing SAP with the correct Location ID**.
				- Later, when SAP capability matures, logic can shift directly to SAP.
				  
				  ---
		- ### **3. Taxation Flow (Vertex Integration)**
			- Important clarifications:
				- Vertex tax calculation requires an **accurate physical address** (state-level granularity).
				- Once XVS receives the Location ID from EUE:
					- XVS requests the full address (including state).
					- XVS sends correct address information to **Vertex**.
					- XVS sends Location ID + Address ID to **SAP** for billing.
				- In the future, Vertex may be integrated directly with EUE (master data single‑source‑of‑truth).
				  
				  ---
		- ### **4. Data Synchronization & Attributes Needed**
			- To support taxation, billing and SAP communication, **two new attributes must be stored at XVS account level**:
				- **Harmonized End‑User Number**
				- **Location ID** (also referred to as address ID)
			- Plan:
				- Run a **one‑time batch sync** for all existing XVS accounts to enrich them with Location IDs.
				- All *new* end customers created in XVS will receive these fields automatically via real‑time sync with EUE.
				  
				  ---
		- ### **5. No Need for Subscription‑Level Changes**
			- Subscription-level address mapping is **not recommended** and should be avoided.
			- All required address information is derived from the **account-level** CMP End‑Customer ID and harmonized Location ID.
			- This significantly simplifies the implementation.
			  
			  ---
		- ### **6. Additional Considerations**
			- RDE report must eventually include:
				- End‑customer address (street, postal code, city, state, country)
				- So resellers can validate which address Ingram used for tax.
			- Multi‑address support is already fully enabled via harmonized EUE.
			- Future SAP upgrade will eliminate dependency on XVS for address resolution.
